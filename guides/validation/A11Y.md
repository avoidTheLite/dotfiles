# Validation — accessibility (a11y)

> **Scope:** The **normative bar** for user interfaces in products that follow these guides. This dotfiles
> repository is mostly Markdown; **automated a11y checks** (ESLint plugin `jsx-a11y`, Playwright, axe) run in
> **application** repositories and in example apps that ship a buildable web UI, not in this guide repo
> unless a concrete example is added under `examples/`.

## Default standard

- Target **[WCAG 2.2](https://www.w3.org/WAI/WCAG22/quickref/)** at **AA** for internal tools, unless a
  specification or legal requirement says otherwise. Treat AAA items as a bonus where cheap.

## What application CI should enforce

- **Lint:** `eslint-plugin-jsx-a11y` (or framework equivalent) on every pull request for React and similar
  stacks, aligned with the [TypeScript client](../client/typescript/STYLE_GUIDE.md) configuration.
- **Runtime checks:** For at least the primary routes or flows, run automated checks in CI (for example
  `@axe-core/playwright` or similar) on a built preview, not only in development.
- **Focus and keyboard:** No critical path may rely on hover-only affordances. Focus order must match visual
  order in common layouts.

## What this guide does not do

- It does not replace a product or legal accessibility audit. It sets a **developer baseline** your TPM or
  spec process can build on.
- It does not mandate a single component library. If you use Shadcn and Radix, document any intentional
  deviation in the [design components](../platform/design/components.md) layer with the standard
  `<!-- guide-exception: a11y.1 reason here -->` pattern where required by your validation scripts.
